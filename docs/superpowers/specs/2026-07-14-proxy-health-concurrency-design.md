# 代理池并发健康检查设计

## 目标

缩短代理池页面“检测全部”的等待时间。当前实现逐个检测代理，每个代理最长等待 8 秒；代理数量增加后总耗时线性增长。本次只优化手动健康检查，不改变注册任务的代理选择、固定出口、故障切换或冷却策略。

## 方案

`ProxyPoolManager.check_all` 使用受控线程池并发检测启用的代理：

- 默认最多 8 个工作线程。
- 实际线程数不超过启用代理数量。
- 禁用代理不发起网络请求，保留其未检测状态。
- 每个任务继续调用现有 `check` 方法，复用当前的超时、延迟、HTTP 状态和冷却记录规则。
- `check_all` 等待全部任务结束后返回完整状态映射，GUI 仍通过现有队列一次性刷新。

不增加新的配置项，避免给普通用户引入不必要的并发参数。

## 数据流

1. 用户点击“检测全部”。
2. GUI 保存当前代理池并禁用检测按钮。
3. 后台线程创建 `ProxyPoolManager`。
4. 管理器筛选启用代理，并以最多 8 个线程并发调用 `check`。
5. 单个代理的成功或失败只更新该代理对应的 `ProxyRuntimeState`。
6. 全部任务结束后，GUI 显示可用数量、延迟和错误，并恢复检测按钮。

## 错误处理

- 单个代理连接失败由 `check` 捕获并记录，不中断其他代理检测。
- 代理池配置无效等管理器级异常继续通过 `proxy_error` 事件反馈，检测按钮必须恢复。
- 空代理池保持现有提示，不创建线程池。
- 并发检测不共享可变网络会话；默认使用 `curl_cffi.requests` 的模块级请求入口，每次调用独立发起请求。

## 兼容性与安全边界

- 不解析 Clash、V2Ray 或其他订阅链接。
- 不按账号主动轮换代理。
- 不在代理全部失效时静默回退直连。
- 不改变 Turnstile、Outlook、CPA 或账号注册流程。
- 不提高注册并发数；并发仅用于用户主动触发的代理连通性检查。

## 测试与验收

- 两个启用代理的检测能够同时进入请求阶段，证明不是串行执行。
- 禁用代理不会发起请求。
- 单个代理失败不会阻止其他代理返回状态。
- 现有代理选择、冷却、脱敏和注册相关测试全部通过。
- Python 编译、PyInstaller 构建和 EXE 启动冒烟测试通过。

## 非目标

- 自动下载或转换代理订阅。
- 自动测试出口地区、匿名度或账号风控评分。
- 在注册过程中同时使用多个代理。
- 通过代理或其他方式绕过人机验证。
