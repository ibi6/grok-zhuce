# Grok Auto Register 稳定性加固设计

## 目标

在不改变注册策略、不提高并发上限、不缩短人工验证等待的前提下，修复当前版本中已验证的并发写入、代理出口、Outlook OAuth 持久化、CPA 超时清理和日志泄密问题，并增加面向正常连通性的安全代理配置中心。

本次采用“稳定性优先”的 B 方案，覆盖确定性缺陷和必要测试，不进行大规模 GUI/CLI 架构重写。

## 范围

### 包含

1. 本地 grok2api Token 池使用跨进程文件锁和原子替换，避免并发写坏或丢失 Token。
2. 相对 Token 文件路径以项目目录为基准解析，支持 `token.json` 这种文件名。
3. 注册浏览器应用 `proxy` 配置；接口代理失败时不再静默直连。
4. 增加代理配置中心，支持直接录入或 TXT 导入 HTTP、HTTPS、SOCKS5 代理。
5. 支持代理健康检测、延迟展示、手动优先选择、失败冷却和任务级故障切换。
6. Outlook OAuth 返回轮换后的 refresh token 时，原子写回原 TXT 或 CSV 凭证文件。
7. CPA 导出增加取消信号；超时后先请求取消并等待清理，禁止后台线程与下一轮浏览器操作并行。
8. GUI 和 CLI 日志不再输出完整邮箱 JWT、refresh token、access token 或密码。
9. `config.json` 使用原子保存；示例配置删除重复键并补齐实际公开配置项。
10. 补充 `paramiko` 运行依赖，使源码环境能够使用 CPA 服务器上传功能。
11. 为以上行为增加自动化测试。

### 不包含

1. 自动处理或绕过 Turnstile、人机验证。
2. 以规避平台风控为目的的代理轮换或提高平台注册并发。
3. 按账号、按请求或按固定时间主动轮换出口。
4. 直接解析 Clash、V2Ray、VLESS、VMess、Trojan 等订阅协议。
5. 大规模重构 3000 多行注册主流程。
6. 改变 CPA chat entitlement 的判定标准。
7. 自动清理服务器中已有的无权限 CPA 文件。

## 设计

### 1. 安全文件写入层

新增小型文件辅助函数，负责：

- 将相对路径解析到项目目录。
- 创建父目录，但正确处理只有文件名、父目录为空的情况。
- 使用同目录临时文件写入、刷新并通过 `os.replace()` 原子替换。
- 对 Token 池的完整“读取—合并—写入”过程使用 `filelock.FileLock`，锁文件放在目标文件旁边。

账号结果文件和纯 Token 追加文件继续采用追加模式，但并发模式下使用对应文件锁，避免多进程写入交叉。

### 2. 代理配置中心与代理语义

`proxy` 表示注册主流程统一代理：临时邮箱 HTTP 请求和 Chromium 注册浏览器都使用它。`cpa_proxy` 仍然优先用于账号初始化、CPA OAuth 和 chat 探测。

Chromium 通过 DrissionPage 的代理配置接收 `proxy`。如果代理不可用，请求应明确失败并记录脱敏后的代理地址，不再自动回退直连。这样可以保证用户看到的配置与真实出口一致。

代理配置中心在 `config.json` 的 `proxy_pool` 数组中保存以下静态字段：

- 代理地址。
- 可选显示名称。
- 是否启用。
- 手动优先级。

最近检测时间、延迟、错误摘要、连续失败次数和冷却截止时间属于运行状态，只保存在内存中，避免重启后使用已经过期的健康结果。

支持的导入格式为每行一个代理地址，空行和 `#` 注释行忽略：

```text
http://127.0.0.1:7897
socks5://127.0.0.1:1080
```

选择规则为：

1. 用户手动指定且检测可用的代理优先。
2. 否则选择已启用、未在冷却期且延迟最低的代理。
3. 任务开始后固定该代理，完整注册流程不主动轮换。
4. 只有发生明确的代理连接错误时，当前代理才累计失败；达到阈值后进入冷却，并为尚未开始的下一次完整尝试选择其他可用代理。
5. HTTP 业务错误、验证码失败、账号权限 403 和 Turnstile 不计为代理故障，也不会触发切换。
6. 所有代理均不可用时任务明确失败，不回退直连。

GUI 在“注册任务”页增加当前任务代理摘要，在“系统设置”或独立代理卡片中提供代理列表、TXT 导入、检测、启用/停用、手动选择和删除操作。空状态、检测中、可用、冷却和错误状态使用统一视觉组件。

Clash/V2Ray 用户可以填写其本机提供的 HTTP/SOCKS5 监听端口；本程序不下载或解析订阅链接。

首个版本只支持无需用户名密码的代理，因为 Chromium 不能直接通过 `--proxy-server` 使用带认证信息的代理 URL。导入认证代理时给出明确错误，不静默丢弃认证信息。日志仍统一脱敏代理 URL，避免后续扩展时泄露凭证。

### 3. Outlook refresh token 持久化

`OutlookAccountPool` 保存凭证文件路径和格式信息。token exchange 返回新 refresh token 时：

1. 更新当前账号对象。
2. 在凭证文件锁内重新读取最新文件。
3. 只替换匹配邮箱的 refresh token。
4. 按原 TXT/CSV 格式原子写回。
5. 更新文件缓存标识，避免同一进程随后重新加载旧状态。

写回失败不隐藏：验证码流程继续使用当前 access token，但账号状态记录持久化失败，并在日志中给出不含 Token 的错误提示。

重复邮箱会在导入时被拒绝，避免池中租用账号与按邮箱查找账号不一致。

### 4. CPA 超时和取消

CPA 导出线程使用独立 `threading.Event`。传入 CPA mint 流程的取消回调同时检查用户停止状态和超时事件。

达到 `cpa_mint_timeout_sec` 后：

1. 设置取消事件。
2. 给予短暂清理窗口并再次 join。
3. 在后台线程真正结束前，不启动下一账号。
4. 如果浏览器调用无法及时响应取消，则关闭当前浏览器使页面操作退出，再完成线程回收。

CPA 结果只有在 chat probe 成功后才允许复制到热加载目录或上传服务器。失败凭证保留 `disabled` 标记用于诊断。

### 5. 日志与错误处理

统一增加凭证脱敏函数，覆盖：

- 邮箱服务 JWT。
- OAuth access/refresh token。
- 密码和 API Key。
- 带认证信息的代理 URL。

GUI、CLI、失败日志和异常预览统一使用该函数。账号输出文件仍按用户现有格式保存完整凭证，因为它是功能产物；界面会继续提示文件敏感。

### 6. 配置和依赖

`config.json` 通过原子替换保存，避免程序退出或磁盘异常留下半截 JSON。读取失败时保留原文件并输出明确错误，不静默覆盖。

`config.example.json` 补充 `concurrency`、浏览器模式和当前 CPA 开关，删除重复 `cpa_proxy`。`requirements.txt` 添加服务器上传实际使用的 `paramiko`。

## 测试

新增或扩展测试覆盖：

1. 20 个并发写入后 Token JSON 仍合法且 Token 不丢失。
2. `token.json` 相对路径能够正常写入项目目录。
3. 代理 TXT 导入能够忽略注释、拒绝非法协议并对地址去重。
4. 代理选择遵循手动优先、健康状态、冷却期和延迟顺序。
5. 普通 HTTP 业务错误不会误触发代理切换。
6. 浏览器选项包含任务选择的代理。
7. HTTP 代理失败不会再次直连请求。
8. 所有代理不可用时返回明确错误。
9. Outlook TXT 和 CSV 都能持久化轮换后的 refresh token，且不改动其他账号。
10. 重复 Outlook 邮箱被拒绝。
11. CPA 超时触发取消并等待线程结束。
12. 日志中不出现原始 Token 或代理密码。
13. 配置原子保存后可重新解析。

完成后运行：

```powershell
python -m pytest -q
python -m compileall -q grok_register_ttk.py modern_ui.py outlook_mail.py cpa_export.py cpa_xai tests
pyinstaller --clean --noconfirm grok_register_ttk.spec
```

并对新 EXE 做启动冒烟测试。

## 验收标准

1. 原有 27 项测试继续通过，新测试全部通过。
2. 并发 Token 池压力测试不再出现 JSON 损坏或 Token 丢失。
3. Outlook refresh token 在进程重启后仍使用最新值。
4. 配置代理时，浏览器和接口均不发生静默直连。
5. 一个完整注册任务保持固定出口，仅在代理连接故障后的下一次完整尝试进行切换。
6. 代理业务错误不会被误判为代理失效。
7. CPA 超时后没有遗留导出线程继续操作浏览器。
8. GUI/CLI 日志不包含完整敏感凭证或代理密码。
9. PyInstaller 构建成功，EXE 能启动。
